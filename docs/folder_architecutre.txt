For **Nomi**, the folder architecture is extremely important because:

* it must support **multiple languages**
* it must support **MCP tools**
* it must support **daemon runtime**
* it must support **plugins later**
* contributors must understand it quickly

So the structure should separate **core engine**, **interfaces**, and **extensions**.

Below is a **production-ready architecture that scales from MVP → large project**.

---

# Detailed Explanation of Each Folder

---

# 1. `nomi/core/`

This is the **heart of the system**.

Everything related to **code context generation** lives here.

```
core/
```

### Responsibilities

* AST parsing
* code unit extraction
* symbol indexing
* dependency graph
* context building

---

## `core/parser/`

Handles **Tree-sitter parsing**.

```
parser/
 ├─ engine.py
 ├─ ast_extractor.py
 └─ node_mapper.py
```

Responsibilities:

* load tree-sitter grammars
* parse files
* extract AST nodes
* convert nodes → CodeUnits

---

## `core/index/`

Symbol indexing system.

```
index/
 ├─ symbol_index.py
 ├─ lookup.py
 └─ search.py
```

Responsibilities:

* map symbols to code units
* fast lookup
* fuzzy symbol matching

---

## `core/graph/`

Dependency graph engine.

```
graph/
 ├─ dependency_graph.py
 ├─ edge_builder.py
 └─ traversal.py
```

Responsibilities:

* build call graph
* detect dependencies
* perform graph traversal

---

## `core/context/`

Context assembly engine.

```
context/
 ├─ context_builder.py
 ├─ context_bundle.py
 └─ resolver.py
```

Responsibilities:

* build minimal context
* retrieve dependencies
* assemble final context bundle

---

## `core/compression/`

Token reduction system.

```
compression/
 ├─ skeletonizer.py
 ├─ token_budget.py
 └─ pruner.py
```

Responsibilities:

* remove unnecessary code
* skeletonize dependencies
* enforce token limits

# Key Architectural Advantages

This structure provides:

### Modularity

Each subsystem is isolated.

### Extensibility

New languages can be added easily.

### Agent Integration

MCP tools are separate.

### Scalability

Future features can be added without refactoring.

---

# One Important Architecture Advice

**Do NOT tightly couple parsing, indexing, and retrieval.**

They must remain separate layers.

Correct layering:

```
Parser
↓
Index
↓
Graph
↓
Context Engine
↓
Agent Tools
```

If these layers are mixed, the system becomes **impossible to scale later**.
