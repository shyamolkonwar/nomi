# Nomi — MVP Architecture Documentation

## 1. Overview

Nomi is a local context-engine infrastructure designed to sit between developer codebases and AI coding agents. Its purpose is to reduce token usage and latency by preventing agents from repeatedly loading full files and repositories into LLM prompts. Instead, Nomi extracts and serves minimal structured code context.

Nomi operates as a background daemon that continuously indexes a repository and exposes context retrieval tools via APIs and agent tool protocols.

The MVP focuses on:

* AST-based code unit extraction
* symbol indexing
* dependency graph construction
* minimal context assembly
* local daemon + tool interface

The system is designed to integrate seamlessly with AI coding agents such as CLI-based assistants, IDE extensions, or autonomous coding tools.

---

# 2. System Goals

## Primary Goals

1. Reduce LLM input tokens by 10–50x
2. Provide minimal relevant code context
3. Operate automatically once installed
4. Support integration with coding agents
5. Maintain real-time repository awareness

## Non-Goals for MVP

The following features are intentionally excluded from the MVP:

* vector databases
* semantic embeddings
* cross-encoder reranking
* distributed indexing
* complex static program slicing

These may be added in future iterations.

---

# 3. High-Level System Architecture

Nomi is composed of five primary subsystems.

1. Repository Watcher
2. Parsing Engine
3. Symbol Index
4. Dependency Graph
5. Context Retrieval Engine

These components operate within a persistent daemon process.

Developer Agents interact with the daemon through tool interfaces.

Architecture Flow:

Repository → File Watcher → Parser → Symbol Index → Dependency Graph → Context Builder → Agent Tool Interface

---

# 4. Core Components

## 4.1 Nomi Daemon

The daemon is the long-running service responsible for maintaining repository state and serving context queries.

Responsibilities:

* initialize repository indexing
* monitor filesystem changes
* maintain symbol index
* maintain dependency graph
* respond to agent requests

Lifecycle:

1. Nomi starts
2. repository scanning begins
3. parser builds initial index
4. daemon opens local API server
5. agents begin requesting context

The daemon runs locally on the developer machine.

---

## 4.2 Repository Scanner

The scanner performs the initial discovery of source files.

Responsibilities:

* traverse project directory
* detect supported language files
* ignore irrelevant directories

Typical exclusions:

* node_modules
* build
* dist
* .git

Outputs:

A list of source files grouped by language.

---

## 4.3 File Watcher

The file watcher ensures Nomi stays synchronized with repository changes.

Responsibilities:

* detect file creation
* detect file edits
* detect file deletion

Upon change:

1. affected file is reparsed
2. symbol index updated
3. dependency graph updated

This ensures context is always current.

---

# 5. Parsing Engine

## Purpose

The parser converts raw source files into structured code units.

The MVP uses incremental parsing via Tree-sitter.

Benefits:

* language-agnostic
* extremely fast
* tolerant to syntax errors

## Parsing Output

Each file is decomposed into structured nodes.

Nodes include:

* function declarations
* class definitions
* method definitions
* imports
* interfaces

Each node becomes a CodeUnit.

---

# 6. CodeUnit Data Model

CodeUnits are the atomic representation of code inside Nomi.

Each unit represents a semantic program component.

Fields:

ID

A unique identifier composed of:

repository path + symbol name

Type

Possible values:

Function
Class
Method
Interface
Module

File Path

Absolute or relative path to source file.

Line Range

Start line and end line of the unit.

Signature

The declaration of the symbol without implementation body.

Body

Full implementation content.

Dependencies

Symbols referenced by the unit.

This representation enables granular context retrieval.

---

# 7. Symbol Index

The symbol index is a searchable mapping of identifiers to CodeUnits.

Purpose:

Quickly locate program entities referenced in developer queries.

Structure:

symbol name → CodeUnit ID

Example:

create_user → CodeUnit(auth/service.py:120-150)

The index allows:

* direct symbol lookup
* prefix search
* file scoped search

Storage:

For MVP the index is stored in SQLite.

Benefits:

* simple
* local
* fast lookups

---

# 8. Dependency Graph

The dependency graph captures relationships between CodeUnits.

Graph Type:

Directed graph.

Nodes:

CodeUnits.

Edges:

Relationships between units.

Edge Types:

CALLS

Function A calls Function B

IMPORTS

Module imports another module

DEFINES

File defines a symbol

IMPLEMENTS

Class implements interface

The graph allows multi-step context retrieval.

---

# 9. Context Retrieval Engine

The context engine assembles minimal code context for AI agents.

It operates in several stages.

Stage 1: Query Interpretation

The developer request is analyzed.

Possible signals:

* explicit symbol name
* file path
* function description

Stage 2: Symbol Resolution

The symbol index is searched.

Best candidate CodeUnits are identified.

Stage 3: Dependency Expansion

The graph is traversed.

Immediate dependencies are added.

Depth is limited to prevent context explosion.

Stage 4: Context Assembly

The engine collects:

* focal CodeUnit
* immediate dependencies

Stage 5: Context Compression

Non-focal dependencies are skeletonized.

This means:

Only signatures and docstrings retained.

Bodies removed.

---

# 10. Context Bundle

The context bundle is the final structured output.

It contains:

Task Context

The developer query.

Focal Code

Full implementation of target symbol.

Dependency Skeletons

Signatures of related symbols.

Repository Map

Optional high-level overview.

The bundle is formatted for LLM prompts.

---

# 11. Agent Tool Interface

Nomi exposes tools that coding agents can call automatically.

These tools return structured context.

Core tools:

get_repo_map

Returns high-level project structure.

search_symbol

Finds symbols matching query.

get_symbol_context

Returns focal code unit.

expand_dependencies

Returns related code units.

Agents orchestrate these tools during reasoning.

---

# 12. Local API Server

The daemon runs a local API server.

Purpose:

Allow external tools to retrieve context.

Endpoints:

GET /repo-map

Returns repository structure.

GET /symbol/{name}

Returns code unit metadata.

POST /context

Returns assembled context bundle.

Responses are JSON structured.

---

# 13. Repository Map

The repo map provides a compact overview of the project.

Example:

Auth

create_user
login_user
validate_token

Payments

charge_card
refund_payment

The map helps agents navigate codebases efficiently.

# 14. Installation and Setup

Developer installs Nomi once.

Initialization:

nomi init

This creates a configuration file.

Configuration File:

.nomi.json

Defines:

languages
ignore patterns
index settings

Starting daemon:

nomi start

Daemon runs automatically in background.

---

# 15. Runtime Workflow

Developer asks coding agent to modify code.

Agent detects Nomi tool availability.

Agent calls:

search_symbol

Nomi returns candidates.

Agent calls:

get_symbol_context

Nomi returns minimal context.

Agent generates code using this context.

Token Reduction:

10x to 50x compared to full file loading.

Context Retrieval Latency:

Under 100 milliseconds.

Indexing Latency:

Under 1 second for 10k LOC repositories.

Incremental Update Latency:

Under 50 milliseconds per file change.

# 18. Summary

Nomi introduces a structural context engine for AI-assisted software development.

By transforming repositories into structured graphs of code units, Nomi enables coding agents to retrieve precise context rather than entire files.

This dramatically improves efficiency, latency, and cost while maintaining correctness.

The MVP focuses on reliability, simplicity, and seamless agent integration.
